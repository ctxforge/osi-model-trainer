# Техническое задание: Образовательная платформа «Сетевые технологии»

**Версия**: 1.0
**Дата**: 2026-02-25
**Базовый проект**: OSI Model Trainer (HTML/CSS/JS, ~9000 строк)
**Методическая основа**: В.Г. Олифер, Н.А. Олифер «Компьютерные сети. Принципы, технологии, протоколы» (5-е издание)

---

## 1. Общее описание

### 1.1 Цель проекта

Трансформация текущего тренажёра модели OSI в полноценную образовательную платформу для изучения компьютерных сетей, протоколов, каналов связи и построения сетей. Платформа должна обеспечить уровень реалистичности, сопоставимый с профессиональными инструментами (MATLAB/Simulink, Wireshark, GNS3), оставаясь при этом доступной через браузер без установки.

### 1.2 Целевая аудитория

- Студенты направлений «Информатика», «Сети и телекоммуникации», «Информационная безопасность»
- Инженеры, готовящиеся к сертификациям (CCNA, CompTIA Network+)
- Преподаватели для проведения лабораторных работ
- Самоучки, изучающие сетевые технологии

### 1.3 Текущее состояние

| Метрика | Значение |
|---------|----------|
| Файлов | 4 (index.html, app.js, data.js, style.css) |
| Строк кода | ~9000 |
| Лабораторных работ | 15 |
| Каналов связи | 12 типов |
| Команд терминала | 21 |
| Отзывчивый дизайн | Мобильный (max-width: 600px), 1 брейкпоинт (768px) |

### 1.4 Известные проблемы текущей кодовой базы

1. **Монолитная архитектура**: app.js — 4993 строки в единой IIFE
2. **Дублирование данных**: в `LAB_EXPERIMENTS` (data.js) повторяющиеся ключи перезаписывают друг друга (fragmentation, routing, ipCalc, tcpHandshake, scenario, devices, tcpVsUdp)
3. **Мёртвый код**: `termScroll.call ? null : null;` (app.js:4965)
4. **innerHTML повсеместно**: потенциальная XSS-уязвимость при пользовательском вводе
5. **Полная перерисовка DOM**: при каждом изменении параметров пересоздаётся всё дерево
6. **Отсутствие десктопной адаптации**: контент ограничен 600px

### 2.3 Паттерн модуля

Каждый модуль экспортирует объект с методами `init()`, `render()`, `destroy()`:

```javascript
// js/labs/physical/signal-generator.js
export const SignalGenerator = {
  init(container, eventBus) { /* подписка на события, создание DOM */ },
  render(params) { /* обновление визуализации */ },
  destroy() { /* отписка, очистка */ },
  getState() { /* для сериализации */ },
  setState(state) { /* восстановление */ }
};
```

### 2.4 EventBus для межмодульного взаимодействия

```javascript
// Вместо прямых вызовов между модулями
eventBus.emit('signal:generated', { samples, sampleRate, params });
eventBus.on('channel:changed', (channelType) => { /* реакция */ });
```

### 2.5 Стратегия вычислений

| Задача | Технология | Обоснование |
|--------|-----------|-------------|
| FFT до 4096 точек | Основной поток, JS | <2 мс, не блокирует UI |
| FFT 4096–32768 точек | Web Audio AnalyserNode | Аппаратно-оптимизированный, бесплатный |
| Тяжёлая свёртка, фильтрация | Web Worker + JS | Не блокирует UI |
| Массовое моделирование канала | Web Worker | Параллельная обработка |
| Генерация сетевых топологий | Web Worker | Графовые алгоритмы могут быть ресурсоёмкими |
| FFT >32768 / реальное время | WASM (KissFFT) в Worker (фаза 3) | ~10-50x ускорение над JS |
| Визуализация спектра | Canvas 2D (OffscreenCanvas в Worker — фаза 3) | Достаточно для 2D-графиков |

## 3. Теоретический раздел

### 3.1 Структура по Олифер & Олифер

Контент организован по частям учебника, каждая тема — интерактивная страница:

#### Часть I. Основы сетей
1. **Эволюция компьютерных сетей** — таймлайн от ARPANET до 5G/6G
2. **Общие принципы построения сетей** — топологии (звезда, кольцо, шина, mesh), коммутация каналов vs пакетов
3. **Архитектура и стандартизация** — OSI vs TCP/IP (существующее, расширить), организации (IEEE, IETF, ITU-T, IANA)

#### Часть II. Физический и канальный уровни
4. **Основы передачи данных** — аналоговые/цифровые сигналы, теорема Найквиста, формула Шеннона, кодирование (NRZ, Manchester, 4B/5B, 8B/10B, PAM-4)
5. **Среды передачи** — витая пара (Cat5e–Cat8), оптоволокно (SM G.652, MM OM1–OM5), радио, коаксиал
6. **Ethernet** — 802.3, формат кадра, CSMA/CD, поколения (10M–400G)
7. **Wi-Fi** — 802.11a/b/g/n/ac/ax/be, CSMA/CA, частоты, модуляция
8. **Коммутация** — мосты, коммутаторы, STP (802.1D), VLAN (802.1Q)

#### Часть III. Сетевой уровень
9. **IP-адресация** — IPv4, CIDR, маски, подсети, IPv6
10. **Маршрутизация** — статическая, RIP, OSPF, BGP, таблицы маршрутизации
11. **Межсетевое взаимодействие** — NAT, DHCP, ARP, ICMP

#### Часть IV. Транспортный уровень и приложения
12. **TCP** — 3-way handshake, конечный автомат (11 состояний), управление потоком (скользящее окно), управление перегрузкой (slow start, congestion avoidance, fast retransmit, fast recovery)
13. **UDP** — дейтаграммы, сравнение с TCP, применение
14. **Прикладные протоколы** — DNS, HTTP/1.1/2/3, FTP, SMTP, POP3, IMAP, SNMP, SSH
15. **Сетевая безопасность** — шифрование, TLS 1.2/1.3, сертификаты X.509, IPsec, firewall

#### Часть V. Управление и QoS
16. **Качество обслуживания** — DiffServ, IntServ, DSCP, очереди
17. **Сетевое управление** — SNMP, мониторинг

### 3.2 Формат каждой темы

Каждая тема содержит:
- **Краткое изложение** (2-3 абзаца)
- **Ключевые определения** (терминология)
- **Схема/диаграмма** (интерактивная)
- **Формулы** (с калькулятором: подставляешь значения — видишь результат)
- **Примеры из реальной жизни** (аналогии)
- **Связь с лабораторной** (кнопка «Попробовать в лаборатории →»)
- **Контрольные вопросы** (самопроверка)

### 3.3 Справочник протоколов

Отдельная подстраница-справочник с поиском и фильтрацией:

| Поле | Описание |
|------|----------|
| Название | HTTP, TCP, IP и т.д. |

Формула затухания в свободном пространстве:
```
FSPL(dБ) = 20·log10(d) + 20·log10(f) + 32.44
```
где d — расстояние в км, f — частота в МГц.

**Модель канала:** каждый тип включает:
- Частотно-зависимое затухание (не линейное!)
- Модель шума (AWGN + импульсный)
- Задержка распространения (расстояние / 2×10⁸ для меди, / 2.1×10⁸ для волокна)
- Джиттер (нормальное распределение)
- BER = f(SNR, модуляция) — кривые реальных модуляций
- Влияние окружающей среды (существующая модель ENV_EFFECTS, расширить)

### 4.4 Лаборатория транспортного уровня (расширение существующего)

#### 4.4.1 TCP конечный автомат

Визуализация всех 11 состояний TCP:

```
CLOSED → LISTEN → SYN_RCVD → ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
CLOSED → SYN_SENT → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
                                 → FIN_WAIT_1 → CLOSING → TIME_WAIT → CLOSED
```

Интерактивно: клик на переход — показывает какой сегмент отправляется/принимается.

#### 4.4.2 Управление перегрузкой TCP (новое)

Анимированная визуализация:

| Фаза | Описание | Визуализация |
|------|----------|-------------|
| Slow Start | CWND удваивается каждый RTT | График CWND vs время |
| Congestion Avoidance | CWND растёт линейно | Продолжение графика |
| Fast Retransmit | 3 duplicate ACK → повтор | Маркер на графике |
| Fast Recovery | ssthresh = CWND/2, CWND = ssthresh + 3 | Продолжение |
| Timeout | ssthresh = CWND/2, CWND = 1 MSS | Падение на графике |

Параметры: MSS (по умолч. 1460), начальный CWND, RTT, вероятность потерь.

Сравнение алгоритмов: Tahoe, Reno, NewReno, CUBIC (по умолчанию в Linux).

#### 4.4.3 Скользящее окно (новое)

Визуализация окна:
- Полоска байтов: отправленные+подтверждённые (зелёный), отправленные+неподтверждённые (жёлтый), разрешённые (синий), запрещённые (серый)

### 5.2 Формат заголовков из RFC

Реалистичные значения полей (из RFC 791, 793, 768):

**IPv4 (RFC 791):**
- Version: 4, IHL: 5 (20 байт), DSCP: 0, ECN: 0
- Total Length: вычисляется реально
- Identification: случайное 16-бит
- Flags: DF=1 (по умолч.), MF=0
- TTL: 64 (Linux) / 128 (Windows) — выбор ОС
- Protocol: 6 (TCP) / 17 (UDP)
- Checksum: вычисляется реально (one's complement)

**TCP (RFC 793):**
- Sequence Number: случайный ISN (32-бит)
- Data Offset: 5 (20 байт без опций)
- Flags: контекстно-зависимые
- Window: 65535 (без масштабирования)
- Checksum: вычисляется с pseudo-header
- Options: MSS (4 байта), Window Scale, SACK Permitted, Timestamps

#### 7.2.2 Механика рандомизации

**Принцип**: квест = шаблон + генератор параметров + валидатор ответа.

```
Шаблон квеста:
{
  id: "3.1",
  title: "IP-адресация",
  template: "Спланируй адресацию для сети {baseNetwork}. Нужно {subnetCount} подсетей: {subnetList}.",
  generate() {
    // Каждый вызов — новые параметры
    const base = randomFrom(["10.0.0.0/16", "172.16.0.0/16", "192.168.0.0/16"]);
    const count = randomInt(2, 6);
    const subnets = generateSubnetRequirements(count); // [{name: "Бухгалтерия", hosts: 25}, ...]
    return { baseNetwork: base, subnetCount: count, subnetList: subnets };
  },
  validate(answer, params) {
    // Проверяет: подсети не пересекаются, хватает хостов, маски правильные
    return checkSubnetting(answer, params);
  }
}
```

**Уровни рандомизации:**

| Уровень | Что меняется | Пример |
|---------|-------------|--------|
| Числовой | IP-адреса, порты, размеры, частоты, SNR | 192.168.**47**.0/26 vs 10.**30.5**.0/24 |
| Структурный | Количество устройств, подсетей, VLAN | 3 подсети vs 6 подсетей |
| Типовой | Тип кабеля, протокол, атака, ошибка | Оптоволокно vs витая пара |
| Топологический | Расположение и связи устройств | Звезда vs дерево vs mesh |
| Контекстный | Сценарий и «история» задачи | «Офис» vs «Завод» vs «Университет» |

**Seed-система**: каждый запуск квеста генерирует seed (число), от которого детерминированно вычисляются все параметры. Seed сохраняется для воспроизводимости (студент может показать преподавателю свой вариант). Два студента с разными seed получают разные задания.

#### 7.2.3 Система оценки квестов

| Критерий | Множитель XP |
|----------|-------------|
| Выполнено без подсказок | ×1.0 (максимум) |
| 1 подсказка | ×0.8 |
| 2 подсказки | ×0.6 |
| 3+ подсказок | ×0.4 |
| Выполнено быстрее среднего | +20% бонус |
| Выполнено с первой попытки | +10% бонус |
| Босс-квест | ×2.0 базовый XP |

#### 7.2.4 Подсказки

Трёхуровневая система:
1. **Направление** (бесплатно): «Обрати внимание на маску подсети» — не даёт ответ, указывает область
2. **Намёк** (−20% XP): «Маска /26 даёт 62 хоста — этого достаточно?» — подталкивает к решению
3. **Решение** (−40% XP): «Для 50 хостов нужна маска /26 (64−2=62 хоста)» — прямой ответ на текущий шаг

Подсказки генерируются из тех же параметров, что и задание (они тоже рандомизированные).

#### 7.2.5 Интерфейс квестового режима

**Экран кампании:**
- Карта глав (1–6) — стилизация под сетевую топологию
- Текущая глава подсвечена, будущие — затемнены
- Звёзды за каждый квест (0–3 звезды в зависимости от оценки)
- Прогресс-бар главы

**Экран квеста:**
- Брифинг: текст задания с контекстом («Коллега Алексей из бухгалтерии жалуется...»)
- Рабочая область: соответствующая лаборатория с предустановленными начальными условиями
- Панель квеста: таймер (информативный, не ограничивающий), подсказки, кнопка «Проверить»
- Результат: звёзды, XP, объяснение правильного решения

#### 7.2.6 Перепрохождение

- Любой пройденный квест можно пройти заново с **новым seed** (новые параметры)
- Повторное прохождение даёт 50% XP (стимулирует, но не позволяет бесконечно фармить)
- Лучший результат сохраняется (звёзды не понижаются)

- «Без подсказок» — пройти главу без единой подсказки
- «Упрямый» — пройти квест с 5-й попытки

**Свободный режим:**
- Теория (прочитал все разделы, ответил на все вопросы)
- Физическая лаб. (все типы модуляции, все коды, все каналы)

## 10. Технические ограничения и решения

### 10.1 GitHub Pages

- **Ограничение**: только статические файлы, нет серверного кода
- **Решение**: все вычисления на клиенте, данные в JS-модулях
- **Размер**: GitHub Pages поддерживает до 1 ГБ, рекомендуется <100 МБ
- **Бюджет**: ~500 КБ JS + ~100 КБ CSS + ~50 КБ HTML + ~50 КБ WASM = ~700 КБ (до gzip)

### 10.2 Без сборщика (zero-build)

- **Модули**: ES Modules (`<script type="module">`) — поддержка 97%+ браузеров
- **Импорт**: `import { SignalGenerator } from './labs/physical/signal-generator.js'`
- **Если нужна минификация**: простой скрипт на esbuild (1 команда) для продакшн-сборки, но необязательно

### 10.3 Совместимость с React Native

Принципы кода, облегчающие будущую миграцию:
1. Логика отделена от DOM (чистые функции для вычислений)
2. Рендеринг через абстракцию (не прямые document.createElement, а render-функции)
3. Canvas-код изолирован (можно заменить на react-native-canvas)
4. Состояние в отдельном модуле (аналог Redux store)

### 10.4 Производительность

| Операция | Бюджет времени | Стратегия |
|---------|---------------|-----------|
| Навигация между разделами | <100 мс | Ленивая инициализация модулей |
| Обновление графика сигнала | <16 мс (60 FPS) | requestAnimationFrame, Canvas 2D |
| FFT 4096 точек | <5 мс | Основной поток |
| Генерация топологии 100 узлов | <1 с | Web Worker |
| Загрузка приложения | <2 с | Критический CSS inline, отложенная загрузка модулей |
